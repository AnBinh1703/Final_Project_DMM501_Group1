\section{Executive Summary}

\subsection{Purpose}
This document specifies the complete end-to-end fraud detection system that:
\begin{itemize}
    \item Scores transactions using machine learning (uncalibrated risk ranking)
    \item Maps scores to decisions using explicit tiered thresholds (LOW/REVIEW/HIGH)
    \item Serves predictions in real-time via FastAPI with observability
    \item Visualizes decisions through an interactive frontend dashboard
    \item Monitors performance with Prometheus metrics and Grafana dashboards
    \item Deploys reproducibly using Docker and Docker Compose
\end{itemize}

\subsection{Key Design Principle}
\begin{quote}
\textbf{\textit{Fraud detection is not classification; it is decision-making under constraints.}}
\end{quote}

The system explicitly separates:
\begin{enumerate}
    \item \textbf{Risk Scoring} (ML model) → uncalibrated ranking signal ∈ [0,1]
    \item \textbf{Decision Intelligence} (Policy layer) → tiered actions bound by operational capacity
\end{enumerate}

\subsection{Core Deliverables}
\begin{itemize}
    \item ✅ Training pipeline with data validation and model selection
    \item ✅ FastAPI prediction service with schema validation
    \item ✅ Prometheus metrics and Grafana dashboards
    \item ✅ Frontend dashboard for real-time decision visualization
    \item ✅ Docker Compose deployment stack
    \item ✅ Unit and integration tests (80\% coverage gate)
    \item ✅ Responsible AI documentation (fairness, explainability, privacy)
\end{itemize}

\section{Problem Definition and Business Context}

\subsection{Business Problem}
Payment card fraud produces:
\begin{itemize}
    \item \textbf{Direct costs}: Chargebacks, reimbursed losses
    \item \textbf{Indirect costs}: Manual review labor, customer support, churn due to friction
\end{itemize}

A fraud scoring service reduces losses by:
\begin{enumerate}
    \item \textbf{Identifying} risky transactions early (before settlement or chargeback)
    \item \textbf{Prioritizing} reviewer capacity (manual checks on highest-value/highest-risk)
    \item \textbf{Enabling} tiered actions (allow low-risk, review medium-risk, block high-risk)
\end{enumerate}

\subsection{Schema}
\begin{itemize}
    \item \texttt{Time} (int): Seconds within transaction period
    \item \texttt{V1-V28} (float): PCA-transformed anonymized features
    \item \texttt{Amount} (float): Transaction amount (currency)
    \item \texttt{Class} (int): 0 = legitimate, 1 = fraud
\end{itemize}

\subsection{Class Distribution (Severe Imbalance)}
\begin{itemize}
    \item Total: 284,807 rows
    \item Legitimate (0): 284,315 (99.83\%)
    \item Fraud (1): 492 (0.17\%)
    \item Fraud rate: ~0.17\% — extremely imbalanced
\end{itemize}

\subsection{Data Validation Pipeline}
\begin{enumerate}
    \item Schema check: Verify columns present and types correct
    \item Missing values: Count and report per column
    \item Duplicates: Flag exact duplicate rows
    \item Statistical summary: Min, max, mean for each feature
    \item Class distribution: Count and log imbalance ratio
    \item Output: artifacts/reports/dataset\_schema.json, missing\_values.csv
\end{enumerate}

\subsection{Data Splitting}
\begin{itemize}
    \item Ratio: 70\% train / 15\% validation / 15\% test
    \item Method: Stratified random (preserves fraud rate)
    \item Stratification column: Class (fraud/non-fraud)
    \item Random seed: 42 (configurable, for reproducibility)
    \item Output: artifacts/reports/split\_info.json
\end{itemize}

\section{Threshold Strategy and Decision Logic}

\subsection{Top-K Capacity-Driven Policy}

Instead of metric optimization, thresholds are set to match operational review capacity:

\begin{table}[H]
\centering
\begin{tabular}{|l|l|r|l|}
\hline
\textbf{Tier} & \textbf{Threshold} & \textbf{Top-K Rate} & \textbf{Condition} \\
\hline
LOW & — & — & score < threshold\_review \\
REVIEW & 0.7391 & 1\% & threshold\_review ≤ score < threshold\_high \\
HIGH & 0.9999 & 0.2\% & score ≥ threshold\_high \\
\hline
\end{tabular}
\end{table}

\subsection{Operational Benefit}

\begin{itemize}
    \item \textbf{Predictable queue size}: If 10K trans/day, expect ~100 review + 20 block cases
    \item \textbf{No data-dependency tuning}: Percentile holds if fraud rate changes
    \item \textbf{Defensible policy}: ``Flag the top 1\% riskiest transactions for review''
\end{itemize}

\subsection{Example}

If system processes 1,000,000 transactions/day:
\begin{itemize}
    \item 10,000 flagged for REVIEW (1\%)
    \item 2,000 flagged for HIGH/BLOCK (0.2\%)
    \item 988,000 allowed automatically
\end{itemize}

\subsection{Decision Logic in API}

Location: \code{src/api/main.py} lines 158-163

\begin{lstlisting}
def _tier_and_action(score, *, threshold_review, threshold_high):
    if score >= threshold_high:
        return "HIGH", "block"
    if score >= threshold_review:
        return "REVIEW", "review"
    return "LOW", "allow"
\end{lstlisting}

\subsection{Risk Score Semantics (Critical)}

\textbf{Claim}: Risk score is \underline{NOT} a calibrated probability of fraud

\textbf{Why}: 
\begin{itemize}
    \item No calibration validation (no Brier score, calibration curves)
    \item Imbalance weighting changes learning objective
    \item Random split (not time-aware) → potential dataset shift
    \item Thresholds are percentile-based (ordinal ranking), not probabilistic
\end{itemize}

\textbf{What it IS}:
\begin{itemize}
    \item Uncalibrated ranking signal
    \item Higher score → higher model confidence
    \item Use for prioritization and tiering, not ``60\% chance of fraud'' claims
\end{itemize}

\textbf{Communicated to users}:
\begin{itemize}
    \item API response includes \code{score\_semantics: "risk\_score\_uncalibrated"}
    \item Frontend disclaimer displayed
    \item Documentation clearly states this limitation
\end{itemize}

\section{Responsible AI}

\subsection{Fairness and Bias}

\subsubsection{Dataset Limitations}
\begin{itemize}
    \item \textbf{Missing protected attributes}: No age, gender, ethnicity, income
    \item \textbf{Implication}: Cannot measure demographic parity or equalized odds
    \item \textbf{Alternative}: Fairness via proxy slices (Amount quantiles, Time buckets)
\end{itemize}

\subsubsection{Proxy Slice Analysis}

\begin{table}[H]
\centering
\small
\begin{tabular}{|l|p{2.5cm}|p{2.5cm}|}
\hline
\textbf{Slice} & \textbf{Definition} & \textbf{Metrics} \\
\hline
Amount Quantiles & Q1, Q2, Q3, Q4 & PR-AUC, precision, recall \\
Time Buckets & Hour-of-day from Time & PR-AUC, precision, recall \\
Combined & Amount × Time & Same across slices \\
\hline
\end{tabular}
\end{table}

\subsection{Explainability}

\textbf{Method}: SHAP (SHapley Additive exPlanations)

\textbf{For Baseline (Logistic Regression)}:
\begin{itemize}
    \item Feature coefficients (linear model)
    \item Output: Coefficient importance table
\end{itemize}

\textbf{For Improved Model (LightGBM)}:
\begin{itemize}
    \item SHAP values
    \item Output: \code{artifacts/figures/shap\_summary.png}
\end{itemize}

\textbf{Key Limitations}:
\begin{itemize}
    \item SHAP is model-level, not causal
    \item Cannot infer ``V5 causes fraud'' from feature importance
    \item Global importance is averaged across all instances
\end{itemize}

\subsection{Privacy}

\textbf{Data Minimization}:
\begin{itemize}
    \item ✅ API accepts feature vector only (no PII)
    \item ✅ Dataset uses anonymized PCA features
    \item ✅ No customer identifiers in artifacts
\end{itemize}

\textbf{Logging Policy (Recommended)}:
\begin{itemize}
    \item ✅ Log: request\_id, model\_version, latency, status
    \item ✅ Log: aggregated metrics (counts, averages)
    \item ❌ Do NOT log: raw feature vectors
    \item ❌ Do NOT log: individual transaction scores
\end{itemize}

\section{Conclusion}

\subsection{System Summary}

This end-to-end fraud detection system demonstrates:

\begin{enumerate}
    \item \textbf{Data-driven decision-making}: Raw transactions → ML scoring → tiered actions
    \item \textbf{Operational awareness}: Capacity-driven thresholds matched to business constraints
    \item \textbf{Observability}: Prometheus metrics and Grafana dashboards for system health
    \item \textbf{Reproducibility}: Versioned artifacts, seeded processes, CI/CD automation
    \item \textbf{Responsible AI}: Fairness analysis, explainability, privacy considerations
\end{enumerate}

\subsection{Key Achievements}

\begin{itemize}
    \item ✅ Baseline vs Improved models: Trained, compared, selected
    \item ✅ Threshold Strategy: Explicit top-K policy  tied to review capacity
    \item ✅ API Design: Clean FastAPI with Pydantic validation
    \item ✅ Frontend: Real-time dashboard with decision visualization
    \item ✅ Monitoring: Prometheus metrics and Grafana dashboards
    \item ✅ Testing: 80\% code coverage with CI/CD automation
    \item ✅ Deployment: Full Docker Compose stack ready for deployment
    \item ✅ Documentation: Comprehensive architecture and design docs
\end{itemize}

\begin{enumerate}
    \item \textbf{Separation of Concerns}: Modular architecture (data, ML, serving, observability)
    \item \textbf{Explainability}: SHAP importance, clear API contracts
    \item \textbf{Accountability}: Version tracking, threshold logging, metric reporting
    \item \textbf{Reproducibility}: Seeded processes, artifact versioning, CI/CD automation
    \item \textbf{Honest Limitations}: Clear docs on uncalibrated scores, production gaps
\end{enumerate}

\textbf{For Demo/Academic Use}: ✅ Complete and rigorous
